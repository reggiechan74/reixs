# REIXS Design Plan (v0.2)

**Real Estate Intelligence Execution Specification**
**Implementation brief for Codex / Claude Code**

## 0) Purpose

### 0.1 The problem

AI-driven real estate workflows (lease abstraction, market rent opinions, acquisition memos) produce outputs that look plausible but fail silently ‚Äî wrong dates, fabricated comps, missing escalation clauses, currency/unit mismatches across jurisdictions. There is no standard way to define what "correct" means for a given task, validate that definition itself, or test outputs against it before they reach a client deliverable.

### 0.2 What REIXS does

REIXS is a **versioned, testable execution specification** for real estate AI workflows. It lets you author a Markdown file that defines:

* What the AI task must accomplish (objective + hard constraints)
* What domain terms mean (DDD references)
* What behaviors are required (SESF rules)
* What "good" looks like (evaluation fixtures)

The system parses, validates, and compiles this spec into machine-readable JSON ‚Äî catching errors in the spec itself before any AI agent runs.

### 0.3 Components

REIXS combines:

* **REMS** (Markdown authoring format)
* **OFD** (Objective Function Design)
* **SESF v3** (Structured English behavior rules; via existing `structured-english` plugin v4.0.0)
* **DDD** (Domain Data Dictionary references)
* **EDD** (Evaluation specs / regression harness)

### 0.4 Immediate use case

**Lease abstraction for Ontario commercial leases.** Given a 40-page lease PDF, REIXS ensures the AI extraction pipeline:
- Extracts all required terms (commencement, expiry, rent, premises area)
- Flags conflicting dates rather than silently picking one
- Marks every extracted value as FACT (verbatim from source) or INFERENCE (derived)
- Uses correct units (CAD, sq ft vs sq m) for Ontario context
- Fails loudly when critical terms are missing rather than inventing them

This is the first vertical slice. If REIXS works for lease abstraction, it works for any structured extraction task.

## 2) Design principles (non-negotiable)

### 2.1 Single source of truth by layer

* REIXS file = task execution contract
* DDD = domain semantics authority
* ADRs = architecture rationale
* EDD = quality thresholds and regression cases
* SESF = behavior rules

Do not duplicate full content across layers.

### 2.2 Machine-checkable over prose

Every ‚Äúimportant‚Äù rule should be testable.
If it cannot be validated, it is guidance, not spec.

### 2.3 Fact/inference separation is first-class

REIXS outputs must support field-level status and provenance.

### 2.4 Hard constraints > score optimization

Any hard constraint violation should fail validation/run readiness.

### 2.5 Version everything

Spec versioning is mandatory from day one.

1. `Meta`
2. `Objective`
3. `Domain Context`
4. `Inputs`
5. `Objective Function Design (OFD)`

7. `Output Contract`
8. `Evaluation / EDD`

* `Spec ID`
* `Version`

* `Secondary Objectives`
* `Tradeoff Policies`

* `Escalation Triggers`

Add under `Domain Context` or a `References` subsection:

### 6.6 Concrete example: what a REIXS file looks like

**Write this FIRST, before coding the parser.** Feel where the format helps vs. creates friction.

```markdown
# REIXS: Lease Abstraction ‚Äî Ontario Commercial

## Meta

- Spec ID: REIXS-LA-ON-001
- Version: 1.0.0
- Task Type: Lease Abstraction
- Tier: standard
- Author: Reggie Chan
- Date: 2026-03-01

## Objective

Extract structured lease terms from a commercial lease document (Ontario jurisdiction)
and produce a normalized term sheet with field-level provenance.

## Domain Context

- Jurisdiction: Ontario, Canada
- Currency: CAD
- Area Unit: sq ft (default), sq m (accepted)
- DDD Reference: re-ddd:lease_core_terms_ontario@0.1.0
- ADR References: ADR-001, ADR-002, ADR-003

## Inputs

- Source document: PDF (scanned or native)
- Document type: Commercial lease agreement
- Expected length: 5-200 pages

## Objective Function Design (OFD)

### Primary Objective
Extract all lease terms defined in DDD with factual accuracy ‚â• 98%.

### Hard Constraints
- NEVER fabricate a term not present in the source document
- NEVER silently resolve conflicting dates ‚Äî flag as CONFLICT
- ALL extracted values must carry provenance (page, clause, verbatim quote)
- Currency must be CAD unless explicitly stated otherwise in source

### AutoFail Conditions
- Any fabricated term (not traceable to source text)
- Missing commencement_date without MISSING status flag
- Provenance absent on any FACT-status field

### Optimization Priority Order
1. Factual correctness
2. Completeness of extraction
3. Provenance quality
4. Formatting consistency

### Uncertainty Policy
When a term is ambiguous or requires interpretation, mark as INFERENCE
with confidence score and reasoning. Never present inferences as facts.

### Secondary Objectives
- Identify renewal options and escalation clauses
- Flag unusual or non-standard provisions

### Tradeoff Policies
- Prefer MISSING over GUESS ‚Äî an empty field with MISSING status is better
  than a plausible but unverified value
- Prefer verbatim extraction over normalized form when normalization is lossy

### Scoring Model
- Weighted F1: precision weight 0.7, recall weight 0.3
- Critical fields (commencement, expiry, rent): 2x weight

### Escalation Triggers
- Confidence < 0.6 on any critical field
- More than 3 fields marked CONFLICT in single document
- Document language is not English

### Error Severity Model
- critical: fabricated term, wrong currency, wrong dates
- high: missing critical field without flag, provenance gap
- medium: formatting inconsistency, missing non-critical field
- low: minor normalization differences, whitespace issues

## Constraints

- Processing time: < 120 seconds per document
- Output format: JSON conforming to lease_abstraction_output_v1.json schema
- No external API calls during extraction (offline-capable)

## Output Contract

Each extracted field must include:
- `value`: the extracted or derived value
- `status`: FACT | INFERENCE | MISSING | CONFLICT
- `provenance`: { page, clause, verbatim_quote } (required for FACT status)
- `confidence`: 0.0‚Äì1.0 (required for INFERENCE status)
- `reasoning`: string (required for INFERENCE and CONFLICT status)

## Evaluation / EDD

- EDD Suite ID: edd:lease_abstraction_core_v1
- Minimum pass rate: 95%
- Zero tolerance: critical-severity errors
- Regression cases: basic_valid_case, conflicting_dates_case, missing_commencement_case

## Behavior Spec (SESF)

```sesf
WHEN extracting lease terms
  FOR EACH field defined in DDD reference
    IF field value found verbatim in source
      THEN status = FACT, provenance = REQUIRED
    IF field value requires interpretation
      THEN status = INFERENCE, confidence = REQUIRED, reasoning = REQUIRED
    IF field not found in source
      THEN status = MISSING, DO NOT fabricate a value
    IF multiple conflicting values found
      THEN status = CONFLICT, list all values with provenance, reasoning = REQUIRED
```

## Validation Checklist

- [ ] All DDD-defined fields addressed in output contract
- [ ] Hard constraints are machine-checkable
- [ ] AutoFail conditions have corresponding test fixtures
- [ ] SESF rules cover extraction, conflict, and missing-value scenarios
- [ ] Ontario jurisdiction metadata (currency, units) declared
- [ ] EDD suite ID references at least 3 test cases
```

## 8) Parser design (Markdown ‚Üí structured object)

### 8.1 Parsing strategy

Use a deterministic parser:

* Parse Markdown headings (`##`, `###`)
* Parse bullet lists and key-value pairs
* Preserve raw text for sections that need freeform content
* Extract fenced code blocks (especially `sesf`)

Do not depend on fragile regex-only parsing for the whole document.

### 8.2 Recommended implementation approach

* Use a Markdown parser library (`markdown-it-py` or `mistune`) or a hybrid line parser
* Convert into an intermediate AST-like structure
* Map headings to canonical section names
* Normalize aliases (`Eval`, `Evaluation`, `Evaluation / EDD`)

### 8.3 Parser error categories

* Missing section
* Duplicate required section
* Invalid heading hierarchy
* Unparseable key-value block
* Missing SESF block
* Unexpected tier value

## 9) Validation design (core of the prototype)

Validation should run in layered passes.

### 9.1 Pass 1: Structural validation

* Required sections present
* Required fields present
* Allowed enum values
* Version format valid
* Tier rules enforced

### 9.2 Pass 2: OFD validation

**Required (all tiers ‚Äî fail if missing):**
* Hard constraints list non-empty
* AutoFail conditions present
* Optimization priority order includes factual correctness / equivalent concept
* Uncertainty policy present

**Required for standard/complex tiers (warn for micro):**
* Tradeoff policy present
* Error severity model includes at least `critical`, `high`, `medium`, `low`
* Scoring model present
* Escalation triggers present

### 9.3 Pass 3: REIXS domain validation

* `Task Type` supported
* `Jurisdiction` declared
* `DDD Reference` present and versioned
* `Output Contract` includes provenance + fact/inference requirement
* `Evaluation / EDD` section references at least one case or suite ID

### 9.4 Pass 4: SESF validation

* Extract `sesf` fenced code blocks from REIXS Markdown
* Write each block to a temp file
* Import and call `parse_sesf()` from vendored `validate_sesf.py`
* Run `check_structural_completeness()`, `check_error_consistency()`, `check_example_consistency()` on parsed result
* Map each `ValidationResult` (category, status, message, line_number) into REIXS report under `sesf_findings`
* REIXS validation FAILS if any SESF result has `status == "FAIL"` (configurable via `--strict-sesf` flag)

### 9.5 Pass 5: Cross-field consistency checks

Examples:

* If `Task Type = Lease Abstraction`, output contract must include normalized lease terms
* If `Tier = complex`, EDD suite ID and ADR refs required
* If OFD hard constraints mention provenance, output contract must enforce provenance
* If Ontario jurisdiction is declared, units/currency should not be missing

### 9.6 Validation output format

Produce both:

* human-readable report (CLI)
* JSON report (`validation_report.json`)

Report schema:

* `status: pass|warn|fail`
* `errors[]`
* `warnings[]`
* `info[]`
* `section_findings{}`
* `sesf_findings[]`

## 16) SESF integration tasks (Day 1: real integration is feasible)

**Updated after reviewing SESF v3 plugin (2026-03-01):** The validator at
`structured-english/skills/structured-english/scripts/validate_sesf.py` is a
~1,480-line pure Python module with zero external dependencies (`re`, `sys`,
`dataclasses`, `pathlib` only). The test file demonstrates direct imports:

```python
from validate_sesf import (
    parse_sesf,
    check_structural_completeness,
    check_error_consistency,
    check_example_consistency,
)
```

This means real SESF validation IS achievable on Day 1. No stub needed.

### 16.1 Validator capabilities (available for import)

The SESF v3 validator provides:

| Function | Returns | What it checks |
|----------|---------|----------------|
| `parse_sesf(filepath)` | `SESFDocument` | Parses SESF markdown into structured object (title, meta, sections, behaviors, procedures, types, functions, actions, precedence) |
| `check_structural_completeness(doc)` | `list[ValidationResult]` | Tier declaration, required sections, meta fields, behavior/procedure presence, rule/step presence, keyword capitalization |
| `check_type_consistency(doc)` | `list[ValidationResult]` | Type references match definitions, orphaned types |
| `check_rule_integrity(doc)` | `list[ValidationResult]` | PRECEDENCE/PRIORITY cross-checks |
| `check_error_consistency(doc)` | `list[ValidationResult]` | ERROR blocks have SEVERITY/ACTION/MESSAGE |
| `check_example_consistency(doc)` | `list[ValidationResult]` | Example coverage per behavior/procedure |
| `check_cross_behavior(doc)` | `list[ValidationResult]` | Complex-tier cross-behavior consistency |

Each `ValidationResult` has: `category`, `status` (PASS/WARN/FAIL), `message`, `line_number`.

### 16.2 Day 1: Integration approach

1. **Copy `validate_sesf.py`** into `reixs/src/reixs/sesf/` as a vendored module (or add `cc-plugins/structured-english` as a git submodule / path dependency)
2. **Extract SESF blocks** from REIXS Markdown (fenced `sesf` code blocks)
3. **Write each block to a temp file** and call `parse_sesf()` + validation functions
4. **Map `ValidationResult` objects** to REIXS validation report format under `sesf_findings`
5. **Fail REIXS validation** if any SESF `ValidationResult` has `status == "FAIL"` (configurable via `--strict-sesf` flag)

### 16.3 SESF v3 features relevant to REIXS

SESF v3 supports both block types that REIXS needs:

* **BEHAVIOR blocks** ‚Üí Declarative rules (e.g., "WHEN conflicting dates found THEN flag as CONFLICT")
* **PROCEDURE blocks** ‚Üí Ordered workflows (e.g., "STEP 1: Extract all date fields from source document")
* **ACTION declarations** ‚Üí Side-effect operations (e.g., "ACTION flag_for_review(field) -> void")

REIXS SESF blocks SHOULD use BEHAVIOR for extraction/validation rules and MAY use PROCEDURE for multi-step workflows.

### 16.4 Important: SESF version alignment

The plugin is at **SESF v3** (plugin version 4.0.0). The REIXS example in ¬ß6.6 uses simple WHEN/THEN syntax which is valid in v3. However, the REIXS validator should accept both:
- Simple WHEN/THEN rules (within BEHAVIOR blocks)
- Full SESF v3 specs with BEHAVIOR/PROCEDURE/ACTION declarations

### 16.5 Future-proofing

Store both:

* raw SESF text
* parsed `SESFDocument` object (the validator already provides this)

This allows later rule execution or transpilation to executable policy graphs.
